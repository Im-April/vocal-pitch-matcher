# vocal-pitch-matcher

# 죠죠 오프닝 음정 채점 웹앱 - 아키텍처 문서

## 1. 프로젝트 개요

- **목표**: 사용자가 죠죠 오프닝 1절 분량을 부르면, 원곡 멜로디와 비교해 음정이 얼마나 맞는지 실시간으로 채점해주는 개인용 웹앱
- **형태**: Streamlit 기반 웹앱 (로컬 실행, 배포 없음 - 개인 소장용)
- **핵심 기능**: 원곡 멜로디 자동 추출 + 실시간 마이크 음정 분석 + 실시간 비교 피드백 + 최종 점수
- **개발 방식**: `streamlit-webrtc`로 처음부터 실시간 스트리밍 방식으로 구현

---

## 2. 전체 구조 (컴포넌트 다이어그램)

```
[오프라인 준비 단계]                    [Streamlit 앱 실행 단계 - 실시간]
─────────────────────                  ──────────────────────────
원곡 음원 (mp3/wav)                     streamlit-webrtc로
      │                                 마이크 스트림 수신
      ▼                                        │
보컬 분리 (Spleeter/Demucs)                     ▼
      │                                 프레임 단위 콜백 실행
      ▼                                (매 오디오 프레임마다 호출)
피치 추출 (librosa pyin / CREPE)               │
      │                                        ▼
      ▼                                피치 추출 (YIN / librosa.pyin)
기준 멜로디 JSON                                │
{time, freq} 배열                       사용자 실시간 Hz
      │                                (st.session_state에 누적)
      └────────────────┬───────────────────────┘
                        ▼
                 실시간 비교 엔진
              (반음 단위 오차 계산)
                        │
                        ▼
              실시간 시각화 + 피드백
           (st.altair_chart 주기적 갱신)
                        │
                        ▼
                   최종 점수 산출
```

## 3. 컴포넌트별 상세 설계

### 3-1. 기준 멜로디 준비 (오프라인, 사전 작업)

**입력**: 죠죠 오프닝 원곡 음원 파일 (개인 소장/로컬 용도)

**처리 파이프라인**:
1. 보컬 분리: Spleeter 또는 Demucs로 반주와 보컬 트랙 분리
2. 피치 추출: 분리된 보컬 트랙에 `librosa.pyin()` 또는 CREPE 모델 적용
3. 후처리: 무음 구간, 노이즈로 인한 이상치(outlier) 프레임 제거/보정
4. 출력 포맷:
```json
[
  { "time": 0.00, "freq": 329.63, "note": "E4" },
  { "time": 0.02, "freq": 330.10, "note": "E4" },
  { "time": 0.04, "freq": null, "note": null }
]
```
   - `time`: 초 단위, 일정 간격(예: 20ms)
   - `freq`: Hz, 무음/휴지 구간은 null
   - `note`: 참고용 음이름 (선택)

**도구**: Python 스크립트 (별도 실행, 웹앱과 분리된 파이프라인)

**주의사항**:
- 반주가 강한 구간은 분리 품질이 떨어질 수 있음 → 필요 시 수작업 보정 구간 확보
- 1절 분량이므로 전체 곡 대비 처리 범위가 작아 관리가 쉬움

---

### 3-2. 웹앱 - 사용자 음정 실시간 추출 (streamlit-webrtc)

**기술 스택**: `streamlit-webrtc` + librosa 또는 경량 YIN 구현

**흐름**:
1. `streamlit_webrtc.webrtc_streamer()`로 오디오 스트리밍 컴포넌트 생성
2. `audio_frame_callback` 함수를 등록 → 마이크에서 오디오 프레임이 들어올 때마다 자동 호출됨
3. 콜백 안에서 프레임 버퍼(numpy 배열)를 받아 피치 추출
   - `librosa.pyin`은 프레임 단위 호출 시 다소 무거울 수 있어서, 짧은 버퍼용으로는 자체 구현한 간단한 **autocorrelation 기반 피치 추출**이 더 가벼울 수 있음 (필요 시 librosa.pyin으로 시작 후 성능 보고 교체)
4. 추출한 Hz 값을 타임스탬프와 함께 `st.session_state`의 리스트에 계속 append
5. 별도 반복 갱신(`st.empty()` + 짧은 주기 rerun, 또는 `streamlit-webrtc`의 상태 콜백)으로 화면을 주기적으로 업데이트

**핵심 난이도 포인트**:
- `streamlit-webrtc`의 콜백은 **별도 스레드**에서 실행되므로, 메인 스레드(화면 렌더링)와 데이터를 주고받을 때 `st.session_state` 접근 시 스레드 안전성 고려 필요 (보통 `threading.Lock` 또는 콜백에서 값만 큐에 쌓고 메인 루프에서 꺼내 쓰는 패턴 사용)
- 화면 갱신 주기가 너무 빠르면 Streamlit이 버벅일 수 있음 → 200~500ms 정도의 폴링 주기로 시작해서 조정 추천

---

### 3-3. 실시간 비교 엔진

**입력**: 
- 기준 멜로디 시계열 `[{time, freq}]` (사전 로드, 앱 시작 시 메모리에 올려둠)
- 사용자 실시간 Hz (콜백에서 매 프레임 들어옴)
- 현재 재생 경과 시간 t (원곡 재생 시작 시점 기준 타이머)

**처리**:
1. 원곡 재생과 마이크 녹음을 동시에 시작 (재생 시작 버튼 하나로 둘 다 트리거)
2. 현재 경과 시간 t에 해당하는 기준 Hz 조회 (가장 가까운 time 매칭)
3. 새로 들어온 사용자 Hz와 비교 → 반음(semitone) 단위 오차 계산
   - 공식: `semitoneDiff = 12 * log2(userFreq / targetFreq)`
4. 오차 범위에 따라 등급 분류 (예: ±0.5반음 이내 = 정확, ±1.5반음 이내 = 양호, 그 이상 = 부정확)
5. 프레임별 점수를 `st.session_state`의 리스트에 누적

**엣지 케이스**:
- 사용자가 무음(숨쉬기 등)일 때 → 채점에서 제외 (피치 추출 결과 신뢰도 낮으면 스킵)
- 기준 멜로디가 null인 구간(간주 등) → 채점에서 제외
- 원곡 재생과 마이크 콜백 간 약간의 지연 발생 가능 → 초반엔 오차 허용 범위를 넉넉히 잡고, 필요하면 수동으로 오프셋(예: 100~200ms) 보정값 추가

---

### 3-4. 시각화 & 실시간 피드백

**화면 구성 아이디어**:
- 기준 멜로디를 미리 전체 그래프로 그려두고, 사용자 음정 라인이 시간에 따라 실시간으로 그려지며 겹쳐짐
- 현재 프레임의 정확도에 따라 실시간 색상 표시 (예: 화면 한쪽에 초록/노랑/빨강 인디케이터)
- 진행 바 (현재 재생 위치 표시)

**구현 방식**:
- `st.empty()`로 플레이스홀더를 만들어두고, 짧은 주기(예: 200~300ms)로 `st.session_state`에 쌓인 데이터를 다시 그려서 갱신
- 차트는 `st.altair_chart` 또는 `st.line_chart` 활용
- 너무 자주 다시 그리면 버벅이므로, 갱신 주기는 실험하면서 조정

**기술 스택**: Streamlit 내장 차트 (`st.altair_chart`) + `st.empty()` 기반 폴링 갱신

---

### 3-5. 최종 점수 산출

- 곡 종료 시점에 프레임별 정확도 누적값을 평균내어 총점(예: 100점 만점) 계산
- 구간별 상세 리포트 (어느 소절에서 많이 틀렸는지) 제공 가능

## 4. 전체 기술 스택 요약

| 영역 | 기술 |
|---|---|
| 전체 | Python + Streamlit (로컬 실행) |
| 원곡 분석 (오프라인, 사전 준비) | librosa (pyin) + Spleeter/Demucs |
| 사용자 음성 입력 | `streamlit-webrtc` (실시간 마이크 스트림) |
| 사용자 피치 추출 | 프레임 콜백 내 librosa.pyin 또는 경량 autocorrelation/YIN |
| 상태 관리 | `st.session_state` (+ 스레드 안전한 큐/락) |
| 데이터 포맷 | JSON 또는 pickle (기준 멜로디 시계열) |
| 시각화 | `st.altair_chart` + `st.empty()` 폴링 갱신 + `st.metric` |

## 5. 예상 난관 및 대응

1. **보컬 분리 품질**: 반주가 센 구간은 오차 가능 → 필요 시 수동 보정
2. **원곡 재생과 녹음 타이밍 동기화**: 원곡을 재생하면서 동시에 녹음을 시작해야 시계열이 잘 맞음 → 앱 안에서 "재생 시작" 버튼과 "녹음 시작" 버튼을 하나로 묶거나, 카운트다운 후 동시 시작하는 UX 고려
3. **마이크 환경 편차**: 마이크 품질/주변 소음에 따라 피치 추출 정확도 편차 발생 가능 → 노이즈 게이트나 신뢰도(confidence) 낮은 프레임 필터링 고려 (librosa.pyin은 voiced_flag를 함께 반환하므로 활용 가능)
4. **저작권**: 개인 소장/로컬 실행 용도로만 사용하고 배포 계획이 없으므로 큰 이슈는 아니지만, 원곡 음원 파일은 리포지토리에 커밋하지 않고 로컬에만 두는 걸 권장
